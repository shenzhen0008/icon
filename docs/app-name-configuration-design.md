# 应用名配置化方案设计稿

## 1. 背景与问题
- 项目已采用 Laravel 标准配置：`config/app.php` 的 `name` 来自 `APP_NAME`，并带默认回退值。
- 当前关注点不在框架能力，而在工程一致性：
  - 需要确保新部署方只改 `.env` 就能完成品牌切换。
  - 需要约束新增文案不得再写死品牌名称。
- 多团队拉取项目后会部署到不同业务场景，期望每个部署环境可快速定义自己的应用名，而无需改业务代码。

## 2. 目标
- 部署方仅通过环境配置即可定义应用名。
- 不引入数据库依赖，不增加首发部署复杂度。
- 与 Laravel 默认实践一致，降低维护成本。
- 统一品牌名称读取入口，减少后续改名成本与回归范围。

## 3. 非目标
- 不在后台页面提供“在线改应用名”功能。
- 不通过数据库保存应用名（避免启动阶段依赖数据库）。
- 不将仓库名、包名、代码内部技术标识强行改为可配置品牌名。

## 4. 设计原则
- Framework First：优先使用 Laravel 原生 `config/app.php + APP_NAME`。
- 配置优先于代码：应用名属于环境配置，而非业务数据。
- 向后兼容：未配置时保留可用默认值。
- 单一读取入口：业务代码统一读取 `config('app.name')`（或基于此的二级配置），不在业务层直接读取 `env()`。

## 5. 方案概述
### 5.1 配置来源
- 主来源：`.env` 中的 `APP_NAME`。
- 读取位置：`config/app.php` 中 `name` 配置：
  - `'name' => env('APP_NAME', 'Zorai Radar')`

### 5.2 模板约定
- 在 `.env.example` 和 `.env.production.example` 中提供统一占位值：
  - `APP_NAME=Your App Name`
- 部署文档要求：首发部署必须显式设置 `APP_NAME`。
- `MAIL_FROM_NAME`、`VITE_APP_NAME` 等派生配置保持引用 `APP_NAME`，避免重复维护品牌名。
- 若应用名包含空格，`.env` 必须加引号，例如：
  - `APP_NAME="Zorai Radar"`

### 5.3 生效机制
- 变更 `APP_NAME` 后，最小必要操作为重建配置缓存：
  - `php artisan config:clear`
  - `php artisan config:cache`
- 若部署流程已统一使用 `optimize:clear/optimize`，可保留，但本文档以配置缓存操作作为基线说明。

### 5.4 范围边界
- 本方案仅替换“用户可见品牌文案”与“由品牌驱动的展示文案”。
- 以下内容默认不在本次配置化替换范围内，除非有明确需求：
  - 仓库名、目录名、包名、命名空间、技术常量。
  - 仅用于内部技术用途的标识（如 hash fallback、本地测试域名等）。
  - 业务测试数据字段（如银行账户默认测试名）。

## 6. 落地改动清单
- 配置模板：
  - `.env.example`
  - `.env.production.example`
- 文档：
  - `README.md`（新增“应用名配置”章节）
- 文案规范：
  - 多语言文案使用 `:app_name` 占位符
  - Blade 渲染统一传入 `['app_name' => config('app.name')]`

## 7. 分阶段实施步骤
### 7.1 第一阶段（低风险，先达成部署可配置）
1. 将 `.env.example`、`.env.production.example` 的 `APP_NAME` 改为统一占位。
2. 更新 `README.md`：新增“部署必填项 APP_NAME”说明与示例。
3. （可选）在 `scripts/deploy.sh` 增加 `APP_NAME` 告警检查（不阻断部署）。
4. 将关键入口（如导航品牌、页面基础标题、邮件发件人）统一改为读取 `config('app.name')` 或其派生配置，不在业务层直接使用 `env()`。
5. 在测试环境执行配置重建并回归关键页面。

第一阶段结果：
- 新环境部署时可通过 `.env` 定义应用名。
- 仍可能存在少量历史硬编码文案（例如多语言字典中的 `Icon Market`）。

### 7.2 第二阶段（中风险，完成全量去硬编码）
1. 扫描并汇总所有硬编码品牌文案（`Icon Market` / `ICON MARKET` / `icon-market`）。
2. 多语言字典统一改造为占位符方案（例如 `:app_name`），并由统一 helper 注入。
3. Blade 中品牌文本改为配置读取，避免模板层硬编码。
4. 同步更新 Feature 测试断言，避免仍断言固定品牌字符串。
5. 全量回归：`php artisan test`、`npm run build`，并做关键页面人工验收。

第二阶段结果：
- 页面、字典、测试与运行时配置一致。
- 部署方仅改 `.env` 的 `APP_NAME` 即可实现全站品牌名称切换。

### 7.3 当前状态（2026-04-20）
1. 第一阶段已完成。
2. 第二阶段核心项已完成（`meta_title`、`notification_title`、`share_text`、导航品牌等）。
3. 当前进入“规范守护”阶段：后续新增文案必须遵循第 8 节规则。

## 8. 验收标准
- 新环境首发部署时，设置 `APP_NAME` 后系统显示为指定名称。
- 不设置 `APP_NAME` 时系统仍可运行（回退默认值）。
- 不需要数据库初始化即可确定应用名。
- 第一阶段验收：
  - `.env` 模板不再使用 `Laravel` 作为示例值。
  - 关键入口页面与系统配置项使用 `config('app.name')` 或派生配置。
  - 执行并通过：`php artisan test`；若涉及前端资源则执行并通过 `npm run build`。
- 第二阶段验收：
  - 用户可见页面/邮件模板/多语言文案不再依赖硬编码品牌字符串。
  - 相关 Feature 测试不再绑定固定品牌字面量，改为配置驱动断言。
  - 执行并通过：`php artisan test` 与 `npm run build`。

## 9. 品牌文案规范（新增）
1. 多语言文件中，凡品牌可见文案统一使用 `:app_name` 占位符。
2. Blade 中渲染翻译时，统一注入：
   - `__('pages.xxx.yyy', ['app_name' => config('app.name')])`
3. 非翻译场景（导航品牌、组件标题）统一使用：
   - `config('app.name')`
4. 禁止新增硬编码品牌文案（例如 `Icon Market`、`Zorai Radar` 字面量用于业务展示）。
5. 测试断言优先使用配置驱动断言，不绑定固定品牌字符串。

## 10. 风险与应对
- 风险：部署方忘记修改 `APP_NAME`，线上仍显示默认值。
  - 应对：部署脚本输出告警 + README 首发清单明确列出。
- 风险：修改后未清理配置缓存，页面未立即生效。
  - 应对：文档固定执行 `config:clear` / `config:cache`（或等价部署流程）。
- 风险：误将内部技术标识当作品牌文案替换，导致兼容性问题。
  - 应对：按“范围边界”执行白名单替换，逐项评审差异。

## 11. 结论
- 采用 `APP_NAME` 环境变量方案即可满足“多部署方自定义应用名”的核心需求。
- 该方案实现成本低、兼容性高，并与 Laravel 官方推荐实践一致。
- 通过两阶段治理，可在不引入新依赖与数据库配置项的前提下，完成品牌名配置化与历史硬编码收敛。
