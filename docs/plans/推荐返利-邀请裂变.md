# 推荐返利（邀请裂变）MVP 业务设计

## 1. 业务目标

建立一个 MVP 版本的两级推荐返利机制：用户通过邀请码注册并绑定上级；下级用户产生正向结算收益后，系统按管理员后台设置的一级、二级比例给上级发放提成。

## 2. MVP 范围

本阶段只实现业务闭环：

- 邀请码注册绑定推荐关系
- 钱包内置浏览器下的邀请码防丢兜底
- 管理员后台设置一级、二级提成比例
- 批处理按已结算收益发放两级提成
- 写入提成记录与余额流水
- 重复执行批处理不重复入账

本阶段不实现：

- 推荐关系审计表
- 推荐关系后台人工修正
- 提成记录后台审计列表
- 配置历史版本
- 邀请码绑定来源统计
- 复杂批次、重试次数、processing 状态机

## 3. 核心业务规则

- 提成对象：一级推荐人、二级推荐人
- 收益来源：下级用户的每一笔已结算收益
- 提成比例：由管理员在后台配置
  - 示例：一级 `5%`、二级 `2%`
- 计算口径：
  - 一级提成：`round(profit * 一级比例, 2)`
  - 二级提成：`round(profit * 二级比例, 2)`
- 发放条件：仅当该笔收益 `profit > 0`
- 资金口径：提成为系统额外发放，不从下级用户收益中扣减
- 生效边界：仅对 `settlement_date >= go_live_date` 的收益生效，不补历史
- 发放方式：管理员或调度任务执行 Artisan 命令批量发放

## 4. 推荐关系口径

- 每个用户有一个自己的 `invite_code`
- 每个用户最多绑定一个直接推荐人：`referrer_id`
- 两级关系示例：`A <- B <- C`
- 当 C 产生正收益时：
  - B 获得一级提成
  - A 获得二级提成
- 当 B 产生正收益时：
  - A 获得一级提成
- 自邀无效
- 无上级关系的层级不发放
- 推荐关系一经绑定不可自动变更
- 已登录用户访问邀请链接不覆盖已有推荐关系

## 5. 邀请码采集口径

- 邀请码来源：`invite_code` query 参数
- MVP 使用三通道防丢：
  - 服务端 session
  - signed cookie
  - 注册表单 `invite_code` hidden input
- 访客访问带邀请码链接时，系统将邀请码同时保存到 session 与 signed cookie
- 页面渲染注册页或注册弹窗时，将当前邀请码写入 `invite_code` hidden input
- 注册提交时优先使用用户提交的 `invite_code`
- 如果注册表单未提交邀请码，则回落读取 session 中的邀请码
- 如果 session 中没有邀请码，则继续回落读取 signed cookie 中的邀请码
- 注册成功且邀请码有效时，写入新用户的 `referrer_id`
- 注册成功后为新用户生成自己的 `invite_code`
- 注册成功后清理 session 与 signed cookie 中的邀请码
- signed cookie 只作为邀请码防丢兜底，不作为信任来源；后端仍需按 `users.invite_code` 校验邀请码有效性

## 6. 后台比例设置

- 管理员可以在后台设置一级、二级提成比例
- MVP 仅维护一条当前配置
- 配置字段：
  - `level_1_rate`
  - `level_2_rate`
  - `is_active`
- 比例校验：
  - `0 <= level_2_rate <= level_1_rate < 1`
- 配置停用时不发放提成
- 提成记录保存当次使用的比例快照，后续修改配置不影响历史记录

## 7. 发放口径

- 发放单位：按“每一笔收益”逐笔计算与发放
- 重复保护：同一笔收益在每个层级只能发放一次
- 提成记录幂等键：`(settlement_id, level)`
- 余额流水业务键：
  - `type = referral_commission_credit`
  - `biz_ref_type = referral_commission`
  - `biz_ref_id = settlement:{settlement_id}:level:{level}`
- 单层发放必须在同一事务内完成：
  - 创建或确认提成记录
  - 锁定获奖用户余额
  - 写入余额流水
  - 更新用户余额
  - 标记提成记录为 `success`
- 发放失败可记录为 `failed`，后续命令可再次尝试

## 8. 时间与计算口径

- 上线边界按业务日期生效：`settlement_date >= go_live_date`
- 业务时区默认使用 `Asia/Shanghai`
- 金额保留两位小数
- MVP 阶段至少保证提成金额计算与余额流水金额一致

## 9. 记录口径

提成记录至少保存：

- 收益记录 ID
- 收益来源用户
- 获奖用户
- 层级
- 基准收益
- 提成比例
- 提成金额
- 发放状态
- 发放时间
- 失败原因

## 10. 验收标准

- 访客通过 `?invite_code=xxx` 进入后注册，成功绑定有效推荐人
- 无效邀请码、自邀不会绑定推荐关系
- 管理员可在后台设置一级、二级提成比例
- 比例不满足 `0 <= 二级 <= 一级 < 1` 时不能保存
- 有推荐关系且收益为正时，上级用户按后台当前比例获得提成
- 下级用户收益不被扣减
- 零收益和负收益不发提成
- 上线前历史收益不会触发提成
- 同一笔收益在同一层级重复执行批处理不会重复提成
- 提成成功后，用户余额、余额流水、提成记录一致
