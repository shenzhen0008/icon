# 推荐返利（邀请裂变）业务设计（两级收益提成版）

## 1. 业务目标

建立两级推荐返利机制：下级用户产生收益时，系统按后台配置的层级比例给上级推荐人发放提成。

## 2. 核心业务规则

- 提成对象：一级推荐人、二级推荐人
- 收益来源：下级用户的每一笔已结算收益
- 提成比例：由管理员在后台配置
  - 示例：一级 `5%`、二级 `2%`
- 计算口径：
  - 一级提成：`round(profit * 一级比例, 2)`
  - 二级提成：`round(profit * 二级比例, 2)`
- 发放条件：仅当该笔收益 `profit > 0`
- 资金口径：提成为系统额外发放，不从下级用户收益中扣减
- 生效边界：仅对功能上线后新产生的收益生效，不补历史

## 3. 推荐关系口径

- 每个用户只绑定一个直接推荐人
- 两级关系示例：`A <- B <- C`
- 当 C 产生正收益时：
  - B 获得一级提成（按后台一级比例）
  - A 获得二级提成（按后台二级比例）
- 当 B 产生正收益时：
  - A 获得一级提成（按后台一级比例）
- 自邀无效（用户不能推荐自己）
- 无上级关系的层级不发放
- 推荐关系一经绑定不可变更（仅允许后台人工审计修正）
- 推荐关系绑定失败时可重试，但必须保证最终只落一条直接上级关系

## 4. 邀请码采集与防丢口径

- 邀请码来源：`invite_code` query 参数
- 邀请码保存必须采用“双通道”：
  - 服务端 session（主通道）
  - 客户端兜底（signed cookie 或注册表单隐藏字段）
- 注册页面加载时先读取 session 邀请码，无则读取 signed cookie，并预填到“推荐人邀请码”输入框
- 用户可以手动修改预填邀请码
- 注册提交时以用户当前输入的邀请码为准；若输入为空，再回落到 session/cookie
- session 失效、跨页面跳转、弹窗注册等场景下，邀请码仍可恢复
- 邀请码仅在账号成功激活并完成绑定后清理
- 已登录用户访问邀请链接不应覆盖或重绑推荐关系
- 邀请码绑定需记录来源：`manual`、`prefill_session`、`prefill_cookie`、`fallback_session`、`fallback_cookie`

## 5. 发放口径

- 发放方式：定时批量发放
- 发放单位：按“每一笔收益”逐笔计算与发放
- 重复保护：同一笔收益在每个层级都只能发放一次
- 发放失败：允许后续重试，但不能导致重复发放
- 幂等键：`(settlement_id, level)`，且同一层级仅允许一个获奖用户
- 状态机：`pending -> processing -> success`，失败进入 `failed` 并可重试
- 发放事务：单层发放必须在同一事务内完成“余额变更 + 流水写入 + 提成记录更新”
- 调度防重入：批处理任务必须加互斥锁，避免多实例并发重复执行

## 6. 时间与计算口径

- 上线边界按“业务日期”生效：`settlement_date >= go_live_date`
- 必须明确业务时区（建议 `Asia/Shanghai`），禁止混用 UTC 时间戳做日期判断
- 提成计算禁止 float 漂移，统一使用 decimal/BCMath（或分为单位的整数）
- 金额保留两位小数，采用统一四舍五入规则并在全链路复用

## 7. 记录与审计口径

- 必须记录“收益来源用户、获奖用户、层级（一级/二级）、提成比例、提成金额、发放时间、发放状态”
- 必须记录提成入账流水，确保资金变化可追溯
- 推荐提成记录与收益记录可关联到同一笔收益来源
- 必须记录批次号、失败原因、重试次数、最后重试时间
- 需要异常审计：同一用户同时出现在同笔收益的多个层级、推荐链路断裂、非法邀请码

## 8. 验收标准（业务视角）

- 有推荐关系且收益为正时，上级用户按后台当前层级比例获得提成（例如一级 5%、二级 2%）
- 下级用户收益不被扣减
- 零收益和负收益不发提成
- 同一笔收益在同一层级不会重复提成
- 上线前历史收益不会触发提成
- 邀请码在“首页进入 -> 产品页弹窗注册/我的页弹窗注册/充值页弹窗注册”链路中不丢失
- 批处理重复执行、多实例并发执行时，不出现重复入账或漏发
- 失败重试后，最终提成记录状态为 `success` 且余额流水唯一
