# 代操盘虚拟币市场网站设计（MVP）

## 1. 目标与范围

### 1.1 目标
构建一个基于 Laravel 13 + Blade + MySQL 5.7 的代操盘展示型网站。用户可注册登录、充值、选择产品池并查看每日已结算收益。

### 1.2 明确不做（本期）
1. 不做交易撮合、不做模拟盘、不做链上交互。
2. 不做提现流程后端能力，仅预留前端按钮。
3. 不做收益承诺，仅展示历史已结算收益。

## 2. 技术与约束

1. 技术栈：Laravel 13、Blade、Tailwind、PHP 8.3、MySQL 5.7。
2. 结构约束：新业务按模块放在 `app/Modules/<Domain>`，避免散乱。
3. 工程约束：遵循项目 `AGENTS.md`（分层、事务、审计、测试与 migration 同步 schema）。

## 3. 功能设计

### 3.1 注册与登录
1. 用户注册/登录。
2. 用户基础资料与账户状态展示。
3. 每个用户使用 21 位用户名（字母区分大小写 + 数字，全局唯一）作为账号标识，正式账号写入 `users.username`。
4. 开发文档统一称“用户名（21位唯一ID）”
5. 认证方案固定为 Laravel 原生 `session + remember me + password.confirm`（其中 `password.confirm` 仅用于已登录用户敏感操作二次验密）。
6. 访问流程：先尝试 `session + remember me` 恢复老用户登录态；若恢复成功则直接按老用户进入，不走临时账号流程。
7. 若未恢复到登录态，则在 `session` 内生成临时账号标识 `temp_username`（21位）；该临时账号仅存在于会话态，不写入数据库。
8. 当访客执行受限操作时弹出“设置密码”；仅在密码创建成功后，才将该账号正式写入 `users` 并转为正式登录账号。
9. 入库时以数据库唯一约束保证 `username` 全局唯一；若写入时发生碰撞（极低概率）则服务端重新生成后重试，直至成功。
10. 本期仅实现基础注册、登录、退出、记住我自动登录、敏感操作二次验密；暂不实现设备绑定、多设备管理、误建合并、增强找回流程。
11. 同设备再次访问优先依赖 `remember me` 自动恢复登录态；目标周期为 7-30 天，默认按 30 天配置。
12. 用户主动退出登录后清除当前设备自动登录态；下次进入需重新登录。
13. 账号身份主依据是用户名 + 密码，不是设备信息。

### 3.2 充值（双通道）
1. 前台自助充值申请：用户提交充值信息，进入待处理。
2. 线下打款人工录入：管理员根据电汇/其他打款记录人工入账。
3. 两种方式统一写入资金流水并可审计。
4. 仅“确认入账”后增加用户本金。

### 3.3 产品池与持仓
1. 后台维护产品池（如 A 产品、B 产品）。
2. 用户选择产品并形成持仓本金记录。
3. 结算严格按“用户所选产品”进行。

### 3.4 每日收益结算
1. 管理员录入产品池每日收益率。
2. 系统按用户持仓所属产品计算当日收益。
3. 示例：用户持有 A 产品，本金 1000，当日 A 收益率 10%，当日收益=100。
4. 只展示历史已结算收益，不展示未来收益承诺。

### 3.5 提现（预置）
1. 前端展示“提现”按钮。
2. 点击后提示“功能建设中”或等价占位提示。
3. 本期不落地提现申请、审核、打款与台账流程。

### 3.6 邀请奖励（老带新）
1. 老用户拥有固定邀请链接（如 `/r/{ref_code}`），可长期分享使用。
2. 新用户通过邀请链接访问后，系统记录邀请来源（cookie/session）。
3. 新用户注册成功后，建立邀请关系（邀请人 -> 被邀请人）。
4. 建立关系后给老用户发放积分奖励，并记录奖励流水。
5. 同一被邀请用户仅允许触发一次邀请奖励（幂等约束）。

## 4. 分层与模块边界

### 4.1 分层职责
1. Controller：请求接收、参数校验、调用服务、响应映射。
2. Service：业务编排（充值确认、收益结算等）。
3. Repository/Query：复杂查询与聚合。
4. Policy/Gate：鉴权与权限边界。
5. Blade：仅展示，不写业务判断和数据访问。

### 4.2 模块建议
1. `app/Modules/User`
2. `app/Modules/Funding`
3. `app/Modules/Product`
4. `app/Modules/Position`
5. `app/Modules/Settlement`
6. `app/Modules/Wallet`

## 5. 数据模型（MVP）

### 5.1 核心实体
1. 用户：`users`（含 `username` 字段，21位唯一账号标识）。
2. 资金账户：`wallet_accounts`。
3. 资金流水：`wallet_ledger`（充值、收益等统一流水）。
4. 充值申请：`deposit_requests`。
5. 产品池：`products`。
6. 产品日收益率：`product_daily_returns`。
7. 用户持仓：`positions`。
8. 日结算明细：`daily_settlements`。
9. 审计日志：`admin_audit_logs`。

### 5.2 关键约束
1. 资金相关变更统一事务提交。
2. 日结算建立幂等唯一键（用户+产品+日期）。
3. 关键字段加索引（用户ID、产品ID、日期、状态）；`users.username` 必须唯一并保证大小写敏感唯一。
4. 不允许负余额与重复入账。
5. 临时账号仅存于 `session`，不落库；只有设密成功后才允许创建 `users` 记录。

## 6. 关键流程

### 6.1 前台自助充值
1. 用户提交申请（pending）。
2. 管理员审核并确认入账。
3. 写 `wallet_ledger` + 更新 `wallet_accounts`。

### 6.2 线下打款人工录入
1. 管理员录入打款凭证与金额。
2. 系统直接执行确认入账流程。
3. 写流水并更新账户。

### 6.3 每日收益结算
1. 录入某产品某日收益率。
2. 扫描该产品有效持仓。
3. 逐用户计算收益并落 `daily_settlements`。
4. 同步写资金流水与账户汇总。

## 7. 风控与安全

1. 所有写操作要求登录与鉴权。
2. 输入统一使用 FormRequest 校验。
3. 关键动作写审计日志（入账、收益录入、结算执行）。
4. 富文本/外部输入按白名单策略处理。

## 8. 测试策略（MVP）

1. 注册与登录：老用户自动恢复登录（session/remember me 命中）、新访客生成 `temp_username`、设密成功后入库、无效参数拒绝。
2. 注册与登录权限：未登录访问受限写接口被拒绝，登录后放行。
3. 敏感操作：未通过 `password.confirm` 拒绝，通过后放行。
4. remember me：勾选记住我后同设备可自动恢复，主动退出后需重新登录。
5. 充值自助：成功、参数错误、未登录。
6. 人工录入：成功、重复凭证拒绝、金额异常拒绝。
7. 产品收益率录入：成功与重复日期策略。
8. 结算：按用户产品计算正确、重复执行幂等。
9. 页面展示：仅显示历史已结算收益。
