# 首页统一动态数值引擎方案

## 目标

首页数值不再各自写一套动态逻辑，而是统一接入一套可复用的动态数值引擎。

这套引擎先服务于首页这两个数值：

1. `Number of people`
2. `总盘获利值`

后续如果首页、列表页、活动页、其他卡片里的数值也要做动态展示，统一继续复用这套引擎，不再重复开发。

## 核心原则

本方案的原则如下：

1. 一个可动态的展示数值，对应一条独立配置
2. 所有动态数值共用同一套服务逻辑
3. 当前值保存在数据库，不保存在前端
4. 用户强制刷新页面后，不会回到初始值
5. 前端只负责显示，不负责决定动态规则
6. 单位、格式、精度规则独立于动态计算逻辑

## 为什么要做统一引擎

如果每个数字自己写一段 JS 或者单独写一个 controller 规则，后面会很快变乱：

1. 规则分散
2. 很难维护
3. 新增一个动态数字就要重新写一遍
4. 刷新、步长、格式、上下限容易不一致

统一引擎的好处是：

1. 所有动态数值共用同一套推进逻辑
2. 后续新增字段只要加配置，不用重写算法
3. 数值当前状态统一保存在数据库，行为可追踪
4. 小流量站点也能保持稳定、简单、可控

## 推荐实现方式

考虑到当前网站访问量很小，可能不到 20 人，这里不建议上复杂缓存方案，也不建议纯前端本地跳值。

推荐采用：

- 数据库保存当前值
- 页面访问时按时间差推进数值
- 前端只读取最终结果

这是当前阶段最稳妥、最好排查、最容易扩展的方式。

## 统一引擎的职责

建议新增统一服务文件，例如：

- `app/Modules/Home/Services/DynamicDisplayValueService.php`

这个服务只负责做以下事情：

1. 根据 key 读取某个动态数值配置
2. 判断是否需要推进数值
3. 如果达到步长时间，则基于当前值推进一次
4. 保存新的当前值和更新时间
5. 返回最新可展示的当前值

它不负责页面文案，不负责前端样式，不负责单位拼接。

## 数据结构建议

建议把“动态展示数值”做成独立表，不和首页普通设置表混在一起。

例如新表：

- `dynamic_display_values`

建议字段如下：

1. `id`
2. `key`
   - 唯一标识
   - 例如：
     - `home_summary_people_count`
     - `home_summary_total_profit`
3. `label`
   - 后台说明名称
4. `base_value`
   - 初始值或参考值
5. `current_value`
   - 当前展示值
6. `min_delta`
   - 每次最小变动值
7. `max_delta`
   - 每次最大变动值
8. `min_value`
   - 允许的最小值
9. `max_value`
   - 允许的最大值
10. `step_seconds`
   - 多久允许推进一次，例如 3 秒
11. `is_dynamic`
   - 是否启用动态变化
12. `format_type`
   - 格式类型，例如：
     - `integer`
     - `money_2`
13. `last_tick_at`
   - 上次推进时间
14. `is_active`
15. `created_at`
16. `updated_at`

## 字段含义说明

### `base_value`

这是后台设定的基础参考值，不一定每次都直接展示，但可以作为系统初始化时的起点。

### `current_value`

这是当前真正展示给用户看的实时值。

页面刷新时，不是重新从 `base_value` 开始，而是从 `current_value` 继续。

### `min_delta` / `max_delta`

表示每次推进时，允许变化的范围。

例如：

- `Number of people`
  - `min_delta = -2`
  - `max_delta = 5`

那就表示每次推进时，人数可能减少 2，也可能增加 5，中间任意整数都可能。

### `step_seconds`

表示每隔多久允许推进一次，例如：

- `3`

意思是：

- 距离上次推进不足 3 秒，不变
- 超过 3 秒，才推进一次

### `format_type`

这个字段不参与动态计算，只决定最终输出格式。

建议先支持两种：

1. `integer`
   - 适合 `Number of people`
   - 千分位
   - 不保留小数
2. `money_2`
   - 适合 `总盘获利值`
   - 千分位
   - 保留两位小数

## 动态推进逻辑

统一引擎建议采用如下逻辑：

### 第一步：读取配置

根据 key 读出对应记录。

例如：

- `home_summary_people_count`
- `home_summary_total_profit`

### 第二步：判断是否启用动态

如果：

- `is_dynamic = false`

则直接返回 `current_value`，不推进。

### 第三步：判断是否达到推进时间

如果当前时间距离 `last_tick_at` 小于 `step_seconds`，则不推进。

如果大于等于，则推进一次。

### 第四步：基于当前值推进

新值不是从 `base_value` 算，而是从 `current_value` 算。

例如：

1. 当前值 `100`
2. 随机增量 `+5`
3. 新值变成 `105`

下次推进时，从 `105` 再继续算，而不是重新从 `100` 算。

### 第五步：限制上下边界

如果新值超出：

- `min_value`
- `max_value`

则截断回允许范围。

### 第六步：保存结果

更新：

1. `current_value`
2. `last_tick_at`

### 第七步：返回格式化结果

根据 `format_type` 返回最终用于页面显示的值。

## 关于强制刷新

本方案的关键目标之一，就是解决“用户强制刷新后数值会不会回到起点”的问题。

答案是：

- 不会

因为当前值保存在数据库里，刷新页面时读取的是数据库中的 `current_value`，不是重新从 `base_value` 计算。

例如：

1. 初始值 `100`
2. 第一次推进到 `105`
3. 第二次推进到 `107`
4. 用户强制刷新

刷新后仍然从 `107` 附近继续，而不是回到 `100`。

## 为什么不用前端本地动态

不建议纯前端本地随机跳值，原因是：

1. 强制刷新后容易回到起始值
2. 不同用户看到的值不一致
3. 页面之间状态不统一
4. 后台无法真正控制当前展示状态

对小站来说，这种方式虽然简单，但不稳。

## 为什么不用缓存方案优先

缓存方案并不是不能用，但对于当前这种小流量网站，不是最稳妥的第一选择。

相较之下，数据库方案更适合当前阶段：

1. 行为直观
2. 排查容易
3. 不依赖额外缓存状态管理
4. 强制刷新行为稳定
5. 后台管理也更自然

后面如果流量变大，再考虑从数据库 current_value 切换到缓存 current_value 也不晚。

## 首页这两个字段的具体规则

### 一、Number of people

建议 key：

- `home_summary_people_count`

规则：

1. 使用 `integer` 格式
2. 后台只填整数基础值和动态参数
3. 页面只显示千分位整数
4. 不显示小数

示例：

- 存储值：`88888`
- 页面显示：`88,888`

### 二、总盘获利值

建议 key：

- `home_summary_total_profit`

规则：

1. 使用 `money_2` 格式
2. 后台只填纯数字
3. 页面显示两位小数和千分位
4. `USDT` 固定写在页面中，不存入动态引擎字段

示例：

- 存储值：`9999999`
- 页面显示：`9,999,999.00 USDT`

## 页面层责任划分

### 后端服务负责

1. 读取动态配置
2. 判断是否推进
3. 推进当前值
4. 保存当前值
5. 返回格式化前的结果或统一格式化后的结果

### 控制器负责

1. 调用统一动态服务
2. 将结果传给 Blade

### Blade 负责

1. 渲染值
2. 固定单位显示
3. 不参与数值推进逻辑

## 后台管理建议

建议做一个统一的后台资源，例如：

- `DynamicDisplayValueResource`

管理员可以在后台配置：

1. 数值名称
2. key
3. 初始值
4. 当前值
5. 最小波动
6. 最大波动
7. 最小边界
8. 最大边界
9. 推进间隔秒数
10. 是否启用动态
11. 格式类型

这样以后新增动态数值时，不需要再改数据库结构，只要新增一条记录即可。

## 推荐实施顺序

建议后续按以下顺序开发：

1. 新建 `dynamic_display_values` 表
2. 新建统一模型
3. 新建统一服务 `DynamicDisplayValueService`
4. 为首页两个 summary 建立配置记录
5. 首页控制器改为通过统一服务取值
6. 前台继续保持当前的格式展示规则
7. 增加后台管理页面
8. 后续其他动态数字接入同一套服务

## 当前结论

本次已确认的方向如下：

1. 首页动态数值不单独写散逻辑
2. 统一做一套动态数值引擎
3. 当前值保存在数据库
4. 推进时以上一次值为基础，不从初始值重算
5. 用户强制刷新不会让数值回到起点
6. 小流量网站优先使用数据库方案，稳定优先
7. `Number of people` 与 `总盘获利值` 先作为第一批接入字段

后续所有类似需求，统一按这套方案扩展。
